> **2026-05-31 XY-cut layout decomposition.** Replaced the flat x-density histogram column detector with a recursive modified XY-cut. Motivation: the old detector failed on layouts where the gutter is interrupted by a wide element (figures straddling both columns, full-width headings/banners) — every page-1-style case in the eval set, and the open carry-forward "column interleaving on page 1" from 2026-05-30.

> - **Types** (`types.rs`): added `Region` / `RegionKind { Leaf, Split }` / `CutAxis`. `ProjectedLine.column_id: usize` → `region_path: Vec<u16>` (index trail from root; equality = same leaf, prefix = containment). `ParsedPage.regions: Region` added behind `#[serde(skip)]` so JSON/text outputs stay byte-identical.
> - **Algorithm** (`projection.rs::xy_cut`): char-weighted 1D projection onto each axis at 2pt buckets. A "valley" is a contiguous run of buckets where density < `T_DENS × median(non-zero buckets)`. Conservative defaults — `T_DENS=0.10`, min vertical-cut valley `8pt`, min horizontal-cut valley `1.4 × median line height`. Each recursion picks the best valley across both axes, tie-breaking 1.1× in favor of horizontal so pages slice into bands before columns (natural reading order). Hard recursion depth cap of 6; H-cuts require ≥2 distinct y-rows per side so we don't slice prose into slivers. Pre-order leaf walk produces reading order for free, including nested layouts.
> - **`build_projected_lines`** now takes `page_height` (additive) and returns `(Vec<ProjectedLine>, Region)`. Lines inside each leaf are y-banded and built with the leaf's `region_path`. `detect_columns` deleted entirely.

> **2026-06-01 paper.pdf quality pass — table fragmentation, cross-region continuation, HR cleanup, figure misclassification.**
>
> - **Borderless-table merged-cell recovery** (`markdown_layout.rs::recover_merged_cell`). PDFium occasionally merges adjacent words into one text run when inter-word kerning is tight (e.g. "MEMORYBANK 5.00 4.77" emitted as one span on `paper.pdf` page 6's results table). That dropped the row's cell count below the established column count, so the table-row matcher closed the run — fragmenting the table into 2-row chunks around every shifted-indent row. Fix: when a candidate row has fewer cells than the established `tracks`, iteratively find the cell that straddles the most tracks (≥2), split its text on whitespace at the boundary positions linearly interpolated to track x-coords, replace with the split pieces. Handles off-by-N (loop), returns `None` cleanly when no whitespace boundary is available (e.g. hyphenated tokens). Added `end_x` to `TableCell`. 3 new unit tests (`recover_merged_cell_splits_off_by_one`, `recover_merged_cell_splits_off_by_two`, `recover_merged_cell_bails_without_enough_whitespace`). **Result on `paper.pdf` page 6:** the 15-row result-table chunks that were previously fragmented into 7+ 2-row pieces are now two coherent 15-row pipe tables (only split by a mid-table "Qwen2.5 GPT Llama 3.2 …" group-label row that's still classified as H3 — header-row absorption remains future work).
> - **Cross-region paragraph continuation** (`markdown_layout.rs::continues_paragraph`). The hard `prev.region_path == cur.region_path` gate broke the common 2-column-paper case where a sentence wraps from the bottom of column 1 into the top of column 2 (e.g. the `- *We present A-MEM …* tem … This system equips LLM agents with` bullet ending mid-thought, and resuming with `long-term interaction capabilities …` in column 2). Relaxed: when `region_path` differs, bridge regions iff the previous line ends without sentence-terminating punctuation (`. ! ? : ; ” " ) ] 。 』 」`) AND the next line starts with a lowercase character. Both conditions are strict signals — together they catch column-wraps cleanly while rejecting unrelated paragraphs that happen to land in adjacent leaves. Smoke on `paper.pdf` collapses ~50 erroneous mid-paragraph breaks (522→479 lines in the rendered markdown).
> - **HR suppression inside / near tables** (`markdown_layout.rs::classify_page_with_filters`). Table internal horizontal rules (row dividers) and the rules drawn above a table's column headers were being emitted as document-level `---`. Build a `table_y_extents` list once after `merge_table_runs`; filter `detect_horizontal_rules` output to drop any HR whose y falls inside `[top - 4 × row_height, bot + 2pt]`. The upward expansion catches header-band rules above the table data that we don't (yet) absorb into the table block. Eliminates the `---` chains that previously surrounded every table on `paper.pdf`.
> - **Figure misclassification of tables** (`figure_cluster.rs::text_coverage`). The old per-glyph area coverage was too tight — left huge gaps between numeric cells unaccounted for, so the page-6 results table (~32% glyph coverage) sneaked under the 55% threshold and got classified as a figure. The figure rect then got obstacle-seeded into XY-cut, which actually helped the rows stay together but the misclassification was waiting to break downstream consumers. Rewrote `text_coverage` to y-band overlapping text items into lines (median-height-based band tolerance) and sum the union bbox of each band — gives a far better proxy for "this region is dominated by text". Result on `paper.pdf`: page 6's table no longer matches as a figure (`figures = []`). No vector figures on `paper.pdf` to regression-test against (page 1's figure is a raster image with 0 vector primitives, so figure_cluster never saw it either way).

> **2026-06-01 step 9c-3 — figure clustering + XY-cut obstacle seeding.** New `crates/liteparse/src/figure_cluster.rs::detect_figure_rects(graphics, text_items, page_w, page_h) -> Vec<Rect>`. Pre-filters HR-like strokes (long thin horizontal, length ≥ 30% page width, ≤ 2pt thick) and full-page rects, then union-finds the remaining primitives by bbox proximity (within 10pt on both axes). Each cluster's union bbox is kept only when: min extent ≥ 30pt both axes, area ≥ 1500pt², not ≥95% page in both dims, aspect ratio ≤ 15, has ≥2 primitives OR a single primitive ≥ 5000pt², and text coverage inside the bbox ≤ 55% (rejects text-highlight backgrounds and banner fills). **Wired through XY-cut as obstacles:** `xy_cut`, `xy_cut_rec`, `xy_find_best_cut`, and `build_projected_lines` now take `figures: &[Rect]`. In `xy_find_best_cut`, each figure intersecting the current bbox is clipped and stamped into the density buckets along the axis being scanned, with `weight = perpendicular extent`. Effect: figures behave like very wide/tall pseudo-items, so the valley search naturally prefers cuts *adjacent* to figures over slicing through them. `figures` is also persisted on `ParsedPage` (#[serde(skip)]) so downstream passes can reuse it. `to_parsed_pages` computes figures from `page.graphics` + `page.text_items` before invoking `build_projected_lines`. **Test count:** 144 lib + 10 integration tests pass (7 new in `figure_cluster::tests` covering empty input, HR rejection, dense-stroke clustering, large-solo-rect acceptance, full-page-background rejection, text-heavy rejection, and distant-strokes-don't-merge; 1 new in `projection::tests::figure_obstacle_forces_h_cut_around_straddling_figure` confirming a wide figure between two text bands forces an H-cut at the root). **Smoke on `paper.pdf`:** pages 2+ unchanged (still clean), abstract column on page 1 still reads cleanly as before. Page 1's title-still-mid-document carry-forward is *unchanged* — the issue isn't the straddling figure (which the obstacle seeding does isolate), it's the arXiv stamp + affiliation lines at the top that satisfy the banner heuristic and disrupt the H-cut position above the title. Banner-detection refinement remains the next layout knob. **Carry-forward update:** the "feeding figure rects into XY-cut as obstacles" item is now resolved.

> **2026-06-01 column-start histogram fallback in XY-cut.** Targets the column-interleaving failure mode behind most remaining MHS=0 docs (decimal section heads on 2-column policy/academic/business reports — docs 029, 038, 039, 077, 189, …). The density-based V-cut search inside `xy_find_best_cut` fails on these layouts because a column-spanning element (wide heading, figure, table) fills the gutter at one y-band, inflating the density at the gutter's x and erasing the valley. Added `xy_find_column_cut` as a *fallback* consulted only when **both** `xy_find_best_cut` calls (H and V) return `None`.
>
> **Algorithm.** Histogram every item's left-edge x with 5pt buckets across the full item span (`min_x..max_x`, not bbox-relative — see the bbox aside below). Cluster contiguous occupied buckets (allowing ≤2-bucket gaps for intra-column drift) into peaks; each peak carries its summed item count. Then apply four conservative gates *in this order*:
> - `XY_COLUMN_MIN_LINES_PER_PEAK = 10` — drop peaks with fewer than 10 items. Indented paragraph-starts and footnote markers produce 3–6 item peaks; real column-left edges have 15+ items in a typical academic page.
> - **Exactly 2 strong peaks remaining.** 3+ peaks means tabular structure (one peak per cell column); we bail rather than slice a table.
> - `XY_COLUMN_PEAK_DOMINANCE = 0.55` — the two peaks together must hold ≥55% of all items. Rules out scattered-x prose layouts (lists, lab manuals, multi-indent docs) where the two tallest peaks are tall only relative to a flat tail.
> - `XY_COLUMN_PEAK_BALANCE_RATIO = 0.4` — smaller-peak / larger-peak count ratio ≥ 0.4. Filters out "single column with one off-x bullet stack" cases (90/10 split → false-positive column cut).
>
> Plus the existing geometric constraint: gap between the two peaks must be ≥ `XY_COLUMN_MIN_GAP_FRACTION = 0.25` × total item-span width. Cut position is the midpoint between the two peaks; returned `CutCandidate` carries `score = 1e9` so it beats every density-derived cut but still loses to `xy_find_banner_cut`'s `f32::INFINITY`.
>
> **Why histogram items, not lines.** The natural first cut was: y-band items into lines, take each line's `x_min`, histogram those. That fails on the *exact* layouts we're trying to fix — items from the left and right columns at the same baseline fold into one "line" whose `x_min` is the left-column edge and whose `x_max` reaches the right-column tail. The right column's left edge is hidden. Switching to per-item left edges lets each column's items contribute to their own peak directly.
>
> **xy_root_bbox aside.** Doc 029 (and a handful of book-spread PDFs) places content at negative user-space x (items at x=-435 with the page reported as 510pt wide). `xy_root_bbox` clamps `min_x.max(0.0)` and caps width to `page_width`, so on those pages half the items live *outside* the returned bbox and `xy_split_bbox` produces a zero-width left child after any V-cut into the negative half. Future work would need a per-page "items extend past MediaBox" detector that conditionally widens the bbox without changing it for normal docs.**

- The Node CLI formats output client-side from `ParseResult`. ✅ resolved 2026-05-31: `result.text` from the napi binding already carries the Rust-formatted markdown (parser's `parse_input` calls `markdown::format_markdown` when `output_format == Markdown` and stuffs the result into `ParseResult.text`). No separate TS-side stub turned out to be in the tree — the note pre-dated the wiring. `node dist/cli.js parse file.pdf --format markdown` and the Python CLI both produce real markdown end-to-end.
- **Step 5/6 sanity check on `paper.pdf`:** headings still detect (`##` / `####` / `#####`), no false code blocks fired on prose (font-name heuristics correctly skip serif/sans body fonts), and no spurious bold paragraphs. No mono content on this paper to exercise the fence path — covered by unit tests instead.
- Smoke test on `paper.pdf` (a two-column arXiv paper): headings (`#`/`##`/`###`/`####`) detect correctly and de-hyphenation joins `prede-/fined → predefined`. Two columns interleave on the same lines — expected until real `column_id` lands (carry-forward from step 2).
- **Tuning that came out of single-column smoke tests (`hard_5.pdf`, `llama2_4pages.pdf`):**
  - PDFium can return `font_size: 1.0` for every glyph when the font size is baked into the text matrix instead of the Tf operator. `build_projected_lines` now falls back to char-weighted `bbox.height` when the histogram size is ≤ 1.5pt — necessary for `llama2_4pages.pdf` to even detect headings.
  - `ProjectedLine.text` carries column-alignment spaces from the grid projection. The paragraph emitter runs `collapse_whitespace` on every line before joining so prose doesn't look like `for    instance`.
  - Justified body text alternates dominant anchors (Left / Right / Floating) line-by-line. Treating any anchor change as a paragraph break shreds normal paragraphs. The break rule now only fires on a *centered* vs *non-centered* flip.
  - Two font-size tolerances: tight (`0.6pt`) for matching the heading map, loose (`2.5pt`) for paragraph continuation, since the height fallback varies a few points based on descenders.
- **Known limitation:** de-hyphenation joins `closed-/source → closedsource`. We can't tell line-break hyphens apart from genuine compound hyphens without a dictionary. pymupdf4llm has the same issue; not blocking.
- **Footnote follow-up:** academic/legal docs often have footnote numbers like `138.` mid-paragraph (`...settlement outcomes.138`) that are real superscripts. They render attached to the preceding word, but the dedicated footnote block at the page bottom is correctly captured as `138. <text>`. The inline marker is informational; not worth splitting in v1.
- `cargo build --workspace` fails on macOS because `liteparse-python` needs to link against Python (build via maturin instead). Use `cargo build -p liteparse -p liteparse-napi` + `cargo check -p liteparse-python -p liteparse-wasm` for local verification.
